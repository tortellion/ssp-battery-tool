# SSP Battery Configuration Tool — Architectural Brief

_Date: 2026-04-16_

---

## What It Does

The SSP Battery Configuration Tool is an internal, web-based engineering sales tool for Solid State Power. Its job is to take a customer's physical and electrical requirements — enclosure dimensions, voltage, capacity, environment, compliance flags — and produce two artifacts:

1. **A viable battery configuration** — specific cell, BMS, thermal, connector, case, and wiring selections drawn from SSP's real component catalog, with series/parallel topology, COGS, and quote price.
2. **A marketing-ready spec sheet** — a structured, print-quality document with comparison table (vs. LFP), proof points, compliance strip, and manufacturing narrative — suitable to hand to a defense OEM like HII or HAVOCAI without engineering review.

Both artifacts are generated by Claude Sonnet 4.6 against a company-controlled system prompt, then validated server-side against a guardrails layer before they ever reach the client. The whole thing runs as a Next.js 14 Pages Router app on Vercel, backed by Supabase (Postgres + Auth).

## What It's Built On

| Layer | Technology |
|---|---|
| Framework | Next.js 14.2 — Pages Router |
| UI | React 18, custom CSS (navy/gold design system) |
| AI | Anthropic Claude Sonnet 4.6 via direct `fetch` to Messages API |
| Database | Supabase (Postgres) — 3 tables, RLS-enforced |
| Auth | Supabase Email/Password + Postgres BEFORE INSERT trigger (domain allowlist) |
| Auth client | `@supabase/ssr` + `@supabase/auth-helpers-react` (SessionContextProvider) |
| Excel import | `xlsx` 0.18.5 (client-side parse in browser) |
| Deployment | Vercel |
| Tests | Vitest — 14 test files covering all sprint findings |

### Database Tables

- **`component_library`** — singleton JSONB blob (one row, whole team); has `previous_data` audit column and `updated_by` ownership gate
- **`configurations`** — history of all generated spec sheets; shared team view; `user_email` denormalized; guardrail-validated before insert
- **`user_settings`** — per-user quote parameters (margin %, labor rate, hours)

### Core API Routes

| Route | Purpose |
|---|---|
| `POST /api/configure` | Rate-limited; server-fetches library; calls Claude; validates schema; whitelists output keys |
| `POST /api/specsheet` | Rate-limited; calls Claude; validates banned phrases + required proof points; returns or 422s |
| `GET\|PUT /api/library` | Singleton library CRUD with audit trail and RLS ownership gate |
| `GET\|POST /api/history` | Shared team history; POST validates spec content before insert |
| `GET\|PUT /api/settings` | Per-user settings with margin clamp |

## Strengths

**1. Guardrails are genuinely layered.**
The banned phrase system, required proof points, and schema validation run server-side on every AI response before it reaches the client. A hallucinated Wh/kg claim or a "zero fire events" statement gets caught and rejected with a 422 — the client never sees it. This is the right architecture for a regulated sales context.

**2. The library is canonical.**
`/api/configure` fetches the component catalog from the database server-side and discards any client-supplied library. The model can only select real components. There is no path from the browser to a fabricated spec sheet citing fictitious cells.

**3. History is append-only and validated.**
Spec content is guardrail-checked before the history INSERT. HistoryTab re-validates on render and shows a warning banner for any record that fails current guardrails. Team members cannot accidentally print a stale or corrupted spec sheet without seeing a flag.

**4. Auth is domain-locked.**
A Postgres BEFORE INSERT trigger on `auth.users` blocks any email address outside `@solidstatepower.com` or `@solidstatemarine.com` at the database layer — not the application layer. No amount of client manipulation bypasses it.

**5. The SVG layout engine is a genuine differentiator.**
`BatteryLayout.jsx` renders a dimensionally accurate top-down and cross-section view of the pack from the model's output — with red/green margin indicators. A salesperson can show this to an OEM and have an honest conversation about fit before any engineering time is spent.

**6. The test suite is a ratchet.**
14 test files cover every sprint finding. The floor only goes up. The guardrail tests are structural (do the API routes actually wire in rate limiting? does `configure.js` fetch library server-side?) so they catch regressions at the callsite, not just the unit.

## Weaknesses

**1. Single-instance rate limiter.**
The per-user sliding-window rate limiter (`lib/rateLimiter.js`) lives in-process memory. On Vercel, each cold-start gets its own counter. A user who hits two different function instances can burst well past 20 requests/60s. Needs Upstash Redis or Vercel KV with atomic increment.

**2. Library ownership gate is soft.**
The RLS UPDATE policy (`updated_by IS NULL OR updated_by = auth.uid()`) means the first person to write the library owns it permanently. If that user leaves, no one else can overwrite it without a manual Supabase dashboard intervention. There is no admin role.

**3. History has no pagination.**
`/api/history` is hardcoded to `.limit(50)`. After the 51st spec sheet, the oldest records silently disappear from the team view. No "load more," no cursor pagination.

**4. Excel import is client-side.**
`xlsx` parsing runs in the browser. A maliciously crafted `.xlsx` (CVE-2023-30533 prototype pollution) executes in the browser context. Migration path: `exceljs` server-side via an API route.

**5. Rate limiter covers only two routes.**
`/api/history` POST and `/api/settings` PUT are not rate-limited. They don't call Anthropic, but burst writes inflate the Supabase row count rapidly.

**6. No structured logging.**
API errors go to `console.error` and appear in raw Vercel function logs. No request ID, no user ID tagging, no latency tracking. Diagnosing why a spec sheet failed guardrails for a specific customer requires manual log archaeology.

**7. `@supabase/auth-helpers-react` is deprecated.**
`SessionContextProvider`, `useUser`, and `useSupabaseClient` still come from the deprecated package. It works today but will not receive security patches. Migration path: `@supabase/ssr` with a custom React context.

## Integration Into the Larger System

### The Object Model

The command model has 19 objects across two groups:

**14 lifecycle objects:**
Touch → Campaign → Asset → Lead → Signal → Qualification → Discovery → **Configuration** → Proposal → Opportunity → Onboarding → Shipment → Ticket → AccountPlan

**5 cross-cutting objects:**
Company · Contact · Validator · QuickBooks · Dashboard

The SSP Battery Tool lives entirely inside the **Configuration** lifecycle object today. It is a standalone configuration engine that produces a Configuration artifact and optionally a Proposal-ready spec sheet.

### Where This Tool Fits

```
Discovery ──► Configuration ──► Proposal ──► Opportunity
                   │
                   ├─ /api/configure  →  Configuration object (topology, COGS, quote)
                   ├─ /api/specsheet  →  Proposal object (spec sheet, comparison table)
                   └─ /api/history    →  AccountPlan (shared team context on a company)
```

The **Configuration** lifecycle object needs three things to become first-class in the system:

1. A **Company** and **Contact** association — the tool captures `company` and `product` as free text today; these need to resolve to canonical records.
2. A **Proposal** output — the spec sheet is already structured JSON; it should create a Proposal object in the system, not just a row in `configurations`.
3. A **QuickBooks** hook — the `calcCOGS` result and quote price are computed client-side today; they should write to QuickBooks as a draft estimate when a Proposal is committed.

---

### Integration Points by Object

**Company / Contact (cross-cutting)**
The tool's `company` and `product` fields are free-text strings today. Integration requires:
- A Company lookup/autocomplete resolving to a canonical Company ID
- A Contact association (who at the OEM requested this configuration?)
- The `configurations` table gains `company_id UUID` and `contact_id UUID` FK columns

**Signal → Qualification → Discovery (upstream)**
The tool has no upstream context today. In the integrated system:
- A **Discovery** record carries enclosure dimensions, voltage, capacity, and compliance requirements *before* Configuration is run — the tool's form pre-fills from the Discovery object
- **Qualification** gates whether a Discovery is ready for Configuration (is the voltage spec locked? are dimensions confirmed?)

**Proposal (downstream)**
The spec sheet JSON (`product_name`, `spec_block`, `comparison_rows`, `proof_points`, `compliance_strip`, `manufacturing`) is already a structured object. In the integrated system:
- A committed spec sheet creates a **Proposal** record with status `draft`
- The Proposal carries COGS, quote price, margin, and the spec content blob
- Proposal lifecycle: `draft → sent → accepted / rejected`

**Opportunity**
When a Proposal moves to `accepted`, it creates an **Opportunity** with the committed configuration as the bill of materials. The `cfg_summary` JSONB (total_cells, cell_id, series/parallel topology) becomes the Opportunity's line items.

**QuickBooks (cross-cutting)**
The COGS calculation today lives in `calcCOGS()` client-side in `index.jsx`. In the integrated system:
- When a Proposal is committed, the system posts a QuickBooks draft estimate via the QuickBooks cross-cutting object
- Line items: `cell.cost_usd × total_cells`, `bms.cost_usd`, `labor = (base_hours + per_cell_hours × total_cells) × labor_rate`
- The QuickBooks object maps SSP component IDs to QB item codes

**Validator (cross-cutting)**
`lib/guardrails.js` is already a Validator. In the integrated system, Validator is a cross-cutting object that applies at multiple lifecycle stages:
- At Configuration: `validateConfigResponse()` — schema and math validation
- At Proposal: `validateSpecSheet()` — banned phrases, required proofs, comparison rows
- At Proposal send: a human-review gate before the spec sheet is emailed to the OEM

The Validator object should be the canonical home for all guardrail logic across the system, not just inside this tool.

**Dashboard (cross-cutting)**
Today the HistoryTab is the only team-facing view. In the integrated system, Dashboard aggregates:
- Configurations generated per week / by vertical / by company
- Proposal win rate (accepted vs. rejected)
- Anthropic API spend by user (the rate limiter already tracks per-user request counts)
- Library staleness (when was `component_library` last updated?)

**AccountPlan**
The `configurations` table is already a shared team view of all customer touchpoints. An **AccountPlan** object in the integrated system is a Company-scoped rollup of all Configurations, Proposals, Opportunities, and Tickets — with the SSP tool's history as one feed into the AccountPlan timeline.

---

### On Performance Management (L10 / Rocks / Scorecard / Issues)

These objects do not exist in this codebase. They belong at the EOS/operational layer of the system, not inside the Configuration tool. In the integrated system, Rocks and Scorecard metrics would be fed *from* Pipeline objects (how many Configurations converted to Proposals, how many Proposals closed as Opportunities this quarter) — not from the tool itself.

The cleanest model: the **Dashboard** cross-cutting object surfaces Scorecard-relevant metrics. A separate **Performance** module (L10, Rocks, Issues) consumes Dashboard data and manages the EOS cadence. The SSP Battery Tool contributes to Dashboard; it has no direct dependency on Performance.
